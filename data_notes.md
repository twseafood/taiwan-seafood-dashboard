# 資料筆記：漁產品交易行情 OpenData（第一階段驗證）

資料期間：2026-06-30 ～ 2026-07-06（民國 1150630 ～ 1150706）
資料筆數：4,062 筆
原始檔：`data/raw/aquatic_20260630_20260706.json`

## 取得方式的一個重要狀況說明

`data.moa.gov.tw` 這個網域對本工具鏈的自動化請求（含 web fetch 與 Claude in Chrome）一律回傳空白，判斷是網站的防護機制擋下非瀏覽器來源的請求；改用同網域下的 `efish.fa.gov.tw` 測試也一樣被擋。最後是你直接用自己的瀏覽器開啟 API 網址、把整頁「列印成 PDF」後上傳給我，我再用 `pdftotext` 還原成 JSON。

還原過程中發現一個技術細節：PDF 文字擷取把「日、魚、頁、心、竹」等約 40 個常用字，錯誤對應成 Unicode「康熙部首」區塊的形似字（例如「魚貨名稱」被存成「⿂貨名稱」）。這是瀏覽器列印PDF時字型對應表的問題，不是政府資料本身有誤；我已經寫好對照表校正回正確的中文字，並確認校正後 4,062 筆全部能正常解析成合法 JSON，往後如果又用「列印PDF」的方式取資料，可以套用同一份對照表（已內建在還原用的腳本邏輯裡）。

之後若能透過瀏覽器擴充功能或改用你自己網路環境直接下載 JSON/CSV，就可以跳過這個轉換步驟。

## 3. 魚種數量

- 依「魚貨名稱」（中文名）去重：**295 種**
- 依「品種代碼」去重：**303 個代碼**

代碼比名稱多 8 個，原因是有 8 個中文名稱各自對應到兩個不同代碼，例如「黑鯛」同時有 1163 與 2015 兩個代碼、「海鱺」有 1166 與 2491、「香魚」有 1191 與 2631 等（完整 8 組已存在下方延伸筆記）。初步觀察 1xxx/2xxx/6xxx 開頭的代碼可能代表不同的分類體系或處理狀態（例如看到 6111「魚內臟」、6188「黑鯊翅【凍】」這類 6xxx 開頭的像是雜項或加工品），這點需要之後參考漁業署的完整代碼對照表才能確認，暫列為待釐清項目。沒有發現同一代碼對應到兩個不同名稱的情況。

## 4. 民國年轉西元的解析方式

`交易日期` 欄位固定是 7 位數字字串 `YYYMMDD`（民國年不補零、月日各兩位），例如 `"1150706"` = 民國115年7月6日 = 西元2026-07-06。換算公式：西元年 = 民國年 + 1911。已在 `scripts/fetch_aquatic_data.py` 內建 `roc_to_ad()` / `ad_to_roc()` 兩個函式，並用這批真實資料驗證過（1150630→2026-06-30、1150706→2026-07-06 皆正確）。

## 5. 缺漏日（休市日）分析

原本設定抓取 7 天（0630～0706），實際只回傳 **5 天**的資料：0630、0701、0702、0703、0706；**0704（週六）與0705（週日）在全部20個市場中完全沒有任何一筆資料**，判斷這兩天所有市場皆休市（或系統當天未開放申報），這是目前看到最明確的規律性缺漏。至於個別市場在平日的零星缺漏（見上方市場清單說明），目前資料量（僅一週）還不足以判斷是休市規律還是單純當天沒有交易，建議之後拉長時間範圍（例如連續4週）再做一次同樣分析以確認規律。

## 6. 4週資料驗證結果（2026-06-09 ～ 2026-07-06，共12,617筆）

因單次 `$top` 上限是 10,000 筆（超過會回傳 `errMsg`），4週資料改分兩段（6/9–6/22、6/23–7/6）下載後合併去重，這次是用瀏覽器「另存新檔」直接存JSON，沒有再出現PDF那種字型亂碼問題。欄位名稱/型別與前一版完全一致。

**休市日規律確認**：28個日曆天中只有17天有資料。缺漏的11天裡，8個週六/週日**全部**缺資料（6/13,14,20,21,27,28,7/4,7/5），驗證了「週末全體休市」是穩定規律。另外多出兩個平日缺資料：6/18（四）、6/19（五）——6/19剛好是端午節（見月曆6月頁「19 端午節/初五」），研判是連假／國定假日休市，而非資料遺漏。也就是說休市規律是「週末＋國定假日」，之後排班或介面設計要把國定假日也算進去，不能只排除週六日。

**市場清單更新為21個**：4週樣本多看到一個先前7天沒出現的市場「興達港」（15筆，17天中只出現2天）。各市場在17個有資料的交易日中的出現率差異很大：基隆、三重穩定在14/17；埔心、台北、新竹等主力市場在13/17；而梓官、澎湖只有8/17，新港5/17，興達港2/17——這幾個小市場很可能不是每個交易日都有進貨/開市，屬於正常的間歇性資料，不是缺漏。

**魚種數量更新**：以「魚貨名稱」去重 **334種**，以「品種代碼」去重 **343個代碼**，同名多代碼的組合從7天樣本的8組增加到 **9組**（新增「其他鯛類」1169/2019）。樣本越大、越有可能再發現新的同名多代碼組合，這點在設計代碼對照表時要注意，不能只靠7天或28天樣本窮舉。

合併後的原始資料存在 `data/raw/aquatic_20260609_20260706.json`。

## 已完成的產出檔案

- `data/raw/aquatic_20260630_20260706.json`：第一次7天原始資料（含 meta 說明取得方式，PDF還原版）
- `data/raw/aquatic_20260609_20260706.json`：4週原始資料，兩段下載合併去重後共12,617筆，用於休市日規律驗證
- `scripts/fetch_aquatic_data.py`：可重複使用的抓取腳本，支援 `--start` `--end` `--type-no` `--top` 參數、自動民國年換算與分頁；但因為 `data.moa.gov.tw` 目前會擋掉本工具鏈的自動化請求，這支腳本尚未在這個環境成功實測，建議你在自己電腦或其他不受該防護機制限制的環境試跑一次確認。
- `data/fish_calendar.json`：由《2026年年年有魚》漁業署月曆逐頁圖片人工判讀整理，12個月、每月主題分類與推薦魚種（中英文名）、當月漁諺、推薦料理。「主要產地」欄位月曆未提供逐魚種資料，暫為null。
- `data/taiwan_regional_species_reference.json`：月曆封底「臺灣附近海域主要經濟性魚種」地圖的概略整理，可作為之後補「主要產地」的參考起點（非官方精確資料，僅供參考）。

## 待你確認的問題 → 已回覆的決策

1. **主要產地**：已依指示用「推薦魚種」中文名關鍵字與 `taiwan_regional_species_reference.json` 做粗略字串比對，自動填入 `fish_calendar.json`。12個月中有8個月比對到區域（例如3月鰺&鯖比對到6個海域、12月旗&鯊比對到南部外洋/東南部外洋），1、4、10月（鱸、觀賞魚、貝）沒有比對到任何區域，維持null。每一筆比對都留了 `主要產地_比對依據`（哪個關鍵字命中哪個區域），方便你之後複核或替換成更準確的資料——畢竟這只是月曆封底地圖的粗略字串比對，不是官方逐魚種產地資料。
2. **品種代碼**：確定第二階段資料庫設計採「分開」處理，同名不同代碼（如黑鯛1163/2015）視為不同紀錄，不合併去重；6xxx開頭的雜項/加工品代碼也維持獨立、不併入一般魚貨分類。
3. **更長期間資料**：已完成，見上方「6. 4週資料驗證結果」——確認休市規律是「週末＋國定假日」，並多發現1個市場與1組同名多代碼。

## 第一階段結論

資料驗證告一段落，可以進第二階段（設計資料庫/網站）了。有兩個技術限制留給下一階段處理：`data.moa.gov.tw` 會擋掉自動化請求，之後不管是排程更新還是部署到 GitHub Pages 的靜態資料，都需要你手動下載或找其他方式取得最新資料，`scripts/fetch_aquatic_data.py` 目前無法在這個環境直接跑通；另外單次 `$top` 上限10,000筆，抓長區間要記得分段下載。
